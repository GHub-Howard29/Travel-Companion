# 程式設計規範

本文件定義 Travel Companion 專案的開發規範，所有新功能及後續維護皆依照本文件執行。

---

# 一、開發原則

## 1. 單一職責（Single Responsibility）

一個檔案只負責一種功能。

例如：

- 一個 Storage 只處理 Storage。
- 一個 Service 只處理 Service。
- 一個 Hook 只處理 Hook。
- 一個 Component 只負責畫面。

避免一個檔案同時負責多項工作。

---

## 2. 模組化設計

新增功能時，優先建立新模組。

避免將所有功能集中於 App.tsx。

App.tsx 僅負責：

- 畫面流程
- State 管理
- 呼叫各模組

---

## 3. Storage 操作規範

程式不得直接操作：

- localStorage
- IndexedDB

所有資料存取皆透過對應 Storage 模組。

例如：

- expenseStorage
- attachmentStorage
- attachmentDb

避免不同地方重複存取資料。

---

## 4. Sync 操作規範

同步流程集中於 Sync 模組。

不得於 Component 或 UI 直接呼叫 Supabase。

所有同步流程統一由 Sync Manager 管理。

---

## 5. Type 管理

所有 TypeScript 型別集中管理。

避免：

- 重複宣告
- 各模組自行建立相同 Type

修改 Type 時，需同步確認所有引用模組。

---

# 二、程式撰寫規範

## 命名原則

檔案：

使用 camelCase。

例如：

- expenseStorage.ts
- attachmentStorage.ts
- syncManager.ts

Component：

使用 PascalCase。

例如：

- ExpenseList.tsx
- ExpenseDialog.tsx

Type：

使用 PascalCase。

例如：

- ExpenseItem
- PendingItem
- TripItem

---

## 函式原則

一個函式只完成一件事情。

避免過長函式。

必要時拆分 Helper Function。

---

## 註解原則

重要流程需加上中文註解。

避免過度註解。

程式本身能清楚表達時，不需額外說明。

---

# 三、Git 開發流程

每個新功能皆建立 Feature Branch。

流程如下：

main

↓

建立 feature 分支

↓

功能開發

↓

npm run build

↓

功能測試

↓

Git Commit（繁體中文）

↓

Merge 至 main

↓

Push

---

Branch 命名建議：

feature/reference

feature/checklist

feature/permission

feature/expense-date

---

Hotfix：

hotfix/pending-badge

hotfix/mobile-upload

---

# 四、Commit 規範

Commit 訊息使用繁體中文。

例如：

新增旅行資訊功能

完成 Checklist 持久化

修正附件同步問題

改善 Pending Badge 更新

避免使用無意義訊息：

update

fix

test

123

---

# 五、功能開發流程

每新增一項功能，依照以下流程：

1. 更新 Roadmap。
2. 討論功能設計。
3. 建立 Feature Branch。
4. 修改程式。
5. 執行 npm run build。
6. 完成功能測試。
7. Git Commit。
8. Merge。
9. 更新文件。

---

# 六、文件維護規範

新增功能前，先確認：

- Roadmap
- 功能規格
- 資料庫設計
- 程式模組說明

功能完成後，更新：

- 版本更新紀錄
- Bug 修正紀錄（如有）
- 功能規格
- Roadmap（更新狀態）

保持文件與程式一致。

---

# 七、設計原則

本專案遵循以下原則：

- Offline First
- 模組化設計
- 單一職責
- 容易維護
- 容易擴充
- 文件與程式同步更新

每新增一項功能，都應避免破壞既有架構。

如需修改既有架構，應先完成設計討論，再開始實作。
