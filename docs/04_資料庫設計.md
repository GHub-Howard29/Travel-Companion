# 資料庫設計

本專案採用「本機優先（Offline First）」架構。

所有資料會先保存於本機，待網路恢復後，再同步至 Supabase。

---

# 系統架構

使用者操作

↓

React State

↓

IndexedDB

↓

Pending Queue

↓

Sync Manager

↓

Supabase Database

↓

Supabase Storage（附件）

---

# Supabase

主要負責：

- 共用帳本資料
- 使用者登入
- 附件儲存
- 雲端同步

---

## expenses

旅行帳本資料。

主要欄位：

| 欄位 | 說明 |
|------|------|
| id | 帳目 ID |
| trip_id | 旅程 ID |
| title | 支出項目 |
| amount | 金額 |
| payer | 付款人 |
| currency | 幣別 |
| attachment_bucket | Storage Bucket |
| attachment_path | 附件路徑 |
| attachment_status | 附件同步狀態 |
| created_at | 建立時間 |
| updated_at | 最後修改時間 |

---

## Storage

負責保存附件。

目前保存：

- 照片
- 收據
- 其他附件

每筆帳目最多對應一筆附件。

---

# IndexedDB

IndexedDB 為主要本機資料庫。

所有主要資料皆先保存於 IndexedDB。

---

## expenses

保存：

- 共用帳本資料
- 個人帳本資料
- 離線新增資料
- 離線修改資料

主要欄位：

| 欄位 | 說明 |
|------|------|
| id | 帳目 ID |
| tripId | 旅程 ID |
| title | 支出項目 |
| amount | 金額 |
| payer | 付款人 |
| currency | 幣別 |
| attachment | 附件資訊 |
| pending | 是否待同步 |
| createdAt | 建立時間 |
| updatedAt | 修改時間 |

---

## expense-attachments

保存本機附件。

主要欄位：

| 欄位 | 說明 |
|------|------|
| id | 附件 ID |
| expenseId | 所屬帳目 |
| tripId | 所屬旅程 |
| blob | 檔案內容 |
| mimeType | 檔案格式 |
| createdAt | 建立時間 |

---

## Pending Queue

保存所有待同步資料。

主要用途：

- 離線新增
- 離線修改
- 離線刪除
- 附件同步

主要欄位：

| 欄位 | 說明 |
|------|------|
| id | Queue ID |
| expenseId | 帳目 ID |
| action | add / update / delete |
| attachment | 是否包含附件 |
| createdAt | 建立時間 |

同步成功後自動移除。

---

# localStorage

保存小型設定資料。

目前包含：

- 使用者設定
- 系統設定
- 上次選擇旅程
- UI 偏好設定

不保存主要帳本資料。

---

# 資料同步流程

新增帳目

↓

React State

↓

IndexedDB

↓

Pending Queue

↓

等待網路

↓

Sync Manager

↓

Supabase Database

↓

Supabase Storage

↓

同步完成

↓

更新 Pending 狀態

---

# 設計原則

1. 採用 Offline First。
2. 所有主要資料先保存於 IndexedDB。
3. 網路恢復後再同步至 Supabase。
4. 帳目與附件分開同步。
5. Pending Queue 統一管理待同步資料。
6. Supabase 為最終雲端保存位置。
