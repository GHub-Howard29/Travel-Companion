# 系統架構

目前系統分為五個層級。

UI
│
▼
React State
│
▼
Storage
│
├── localStorage
├── IndexedDB
└── Attachment Storage
│
▼
Sync Manager
│
▼
Supabase

---

## UI

負責畫面顯示與使用者操作。

主要包含：

- 共用帳本
- 個人帳本
- 新增／編輯帳目
- 附件管理
- 同步狀態
- 待同步數量（Pending Badge）

---

## React State

負責目前畫面的資料管理。

主要包含：

- 帳目資料
- 使用者資訊
- 同步狀態
- 篩選條件
- UI 狀態
- Pending 狀態

React State 只負責畫面資料，不直接負責資料永久保存。

---

## Storage

負責所有本機資料保存。

### localStorage

保存：

- 使用者設定
- 系統設定
- 目前選擇的帳本
- 其他小型設定資料

---

### IndexedDB

保存：

- 共用帳本
- 個人帳本
- 帳目資料
- Pending Queue
- 附件資訊
- 離線資料

所有主要資料皆優先保存於 IndexedDB。

---

### Attachment Storage

負責附件管理。

包含：

- 新增附件
- 修改附件
- 刪除附件
- 本機附件保存
- Supabase Storage 同步

附件與帳目資料分開管理，以降低同步失敗率。

---

## Sync Manager

負責本機與雲端同步。

主要工作：

- 上傳待同步資料
- 下載最新資料
- Pending Queue 管理
- 衝突處理
- 附件同步
- 同步完成後更新狀態

同步採用 Offline First 設計。

沒有網路時仍可正常操作。

恢復網路後自動同步。

---

## Supabase

負責雲端資料保存。

包含：

- Database
- Storage
- Authentication

提供：

- 共用帳本
- 雲端附件
- 使用者登入
- 雲端同步

---

## 系統資料流程

新增或修改帳目：

使用者操作

↓

React State 更新

↓

IndexedDB 儲存

↓

加入 Pending Queue

↓

等待同步

↓

Sync Manager

↓

Supabase

↓

同步完成

↓

更新 Pending 狀態

---

## 系統設計原則

1. 採用 Offline First 設計。
2. 所有主要資料優先保存於本機。
3. 網路恢復後自動同步。
4. 帳目與附件分開同步。
5. 保持模組化設計，降低模組間耦合。
6. 每個模組只負責單一工作。
